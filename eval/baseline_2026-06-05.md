# Mizan Search-Quality Baseline — 2026-06-05

> The owner's complaint was that "the search quality is very low." This document
> converts that into measured numbers that every future ranking change must beat.
> All figures come from `eval/run_eval.py` run against the **live production API**
> (`https://mizan-api.rollingcatsoftware.com`) on 2026-06-05, with reranking in
> its current prod configuration (`ENABLE_RERANKING=true`, English-only
> `cross-encoder/ms-marco-MiniLM-L-6-v2`).

## How this was measured

- Harness: `python eval/run_eval.py` (stdlib-only; no extra deps).
- Labelled set: `eval/queries.json` — **24 cases**, 8 each in EN / TR / AR,
  arranged as cross-lingual triples of the same concept (mercy, patience,
  charity, forgiveness, prayer, oneness-of-God, light, supplication). Labels are
  conservative high-precision anchors (e.g. *light* → 24:35 Ayat an-Nur), so
  precision@k and MRR stay meaningful on a small set.
- Metrics: precision@10, recall@10, Mean Reciprocal Rank (MRR), by language and
  overall. Only `source_type == "QURAN"` results are scored (the labels are
  surah:verse references).
- Commands (reproducible):

  ```bash
  python eval/run_eval.py --json                 # current prod (rerank on)
  python eval/run_eval.py --rerank false --json  # raw RRF (rerank off) A/B
  ```

## Baseline numbers — current prod (rerank ON, ms-marco English)

| lang | P@10  | R@10  | MRR   | n  |
|------|------:|------:|------:|---:|
| ar   | 0.238 | 0.265 | 0.442 | 8  |
| en   | 0.268 | 0.313 | 0.525 | 8  |
| tr   | 0.178 | 0.236 | 0.469 | 8  |
| **ALL** | **0.228** | **0.271** | **0.478** | **24** |

## A/B — same harness with reranking OFF (raw RRF)

| lang | P@10  | R@10  | MRR   | n  |
|------|------:|------:|------:|---:|
| ar   | 0.238 | 0.265 | 0.442 | 8  |
| en   | 0.172 | 0.208 | 0.396 | 8  |
| tr   | 0.181 | 0.194 | 0.421 | 8  |
| **ALL** | **0.197** | **0.222** | **0.419** | **24** |

## The AR/TR gap (the headline finding)

Compare the two tables **per language**:

| lang | MRR rerank-off | MRR rerank-on | Δ MRR from reranking |
|------|---------------:|--------------:|---------------------:|
| en   | 0.396 | 0.525 | **+0.129** |
| tr   | 0.421 | 0.469 | +0.048 |
| ar   | 0.442 | **0.442** | **+0.000** |

- **Arabic gets ZERO benefit from reranking** — every AR metric is
  byte-identical with reranking on and off (P@10 0.238, R@10 0.265, MRR 0.442 in
  both runs). The current reranker only re-scores candidates that carry an
  English `translation_text`; raw Arabic verse/keyword hits are never fed to it,
  so they stay pinned at the RRF floor. This is the measured root cause of "low
  search quality" for Arabic.
- **Turkish barely benefits** (MRR +0.048) for the same reason — only the EN
  translation of a TR-matched candidate is ever reranked, never the Turkish text.
- **English is the only language reranking actually helps** (MRR +0.129), which
  is exactly why English looks acceptable while AR/TR feel weak.

Language-aware reranking (merged in PRs #16/#17) feeds native Arabic/Turkish
candidate text to a *multilingual* reranker — but that code path only activates
when a multilingual reranker is loaded. Prod still runs English-only ms-marco, so
AR/TR are unchanged until `RERANKER_MODEL=jinaai/jina-reranker-v2-base-multilingual`
is enabled (operator go/no-go in `docs/RUNBOOK.md §0.1`).

## Notable per-case misses (worth labelling/diagnosing later)

- `forgiveness` / `bağışlama` (EN/TR) miss entirely (RR=0). The Arabic `مغفرة`
  scrapes RR=0.200. Forgiveness vocabulary is diffuse across the corpus.
- `light` / `nur` / `نور` miss in **all three** languages (RR=0) despite 24:35
  (Ayat an-Nur) being an unambiguous anchor — a concrete recall failure to chase
  once the multilingual reranker is on.
- `dua` / `دعاء` (TR/AR) miss while EN `supplication` scores RR=0.500.

These are recorded as future targets; the metric exists precisely so a fix can be
proven rather than guessed.

## Gate for future changes

Any ranking/model change must be re-run with the same two commands and **must not
regress** the ALL-language MRR below the rerank-on baseline of **0.478**, and
specifically must **lift AR MRR above 0.442** (the whole point of activating a
multilingual reranker). Use `--min-mrr` to enforce a floor in CI:

```bash
python eval/run_eval.py --json --min-mrr 0.45   # fails build if ALL MRR < 0.45
```
